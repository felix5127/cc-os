# CC-GEB — 项目配置与文档治理

## 使用方式

```
/cc-geb              自动检测状态，推荐动作
/cc-geb init         初始化项目配置
/cc-geb check        文档与代码一致性巡检
/cc-geb upgrade      升级配置级别（轻量→标准→完整）
```

也可以自然语言触发："帮我初始化项目"、"检查文档"、"升级到完整模式"。

---

## 路由逻辑

收到 `/cc-geb` 后，先检测项目状态，再路由到对应子功能：

```
/cc-geb（无参数）
  │
  ├── 没有 CLAUDE.md → 建议 init
  ├── 有 CC-GEB CLAUDE.md → 提示当前级别，可 check 或 upgrade
  └── 有非 CC-GEB CLAUDE.md → 提示可 check，或补充 STACK.md/workflow/

/cc-geb init    → 进入 Init 流程
/cc-geb check   → 进入 Check 流程
/cc-geb upgrade → 进入 Upgrade 流程
```

---

## Init：初始化项目配置

### Step 1: 检测项目状态

```
检测项：
- git 仓库？(git rev-parse --is-inside-work-tree)
- 项目根目录？(git rev-parse --show-toplevel，否则用 pwd)
- 已有 CLAUDE.md？内容是否包含 CC-GEB 标记？
- 已有 .claude/ 目录？
- 包管理文件？(package.json / go.mod / pyproject.toml / Cargo.toml / pom.xml)
- 源文件数量？(粗略统计)
- 目录结构？(顶层目录)
```

### Step 2: 判断场景

```
(A) 没有 CLAUDE.md
    → 进入 Step 3 生成配置

(B) 有 CLAUDE.md，不是 CC-GEB 生成的
    → 告知"检测到已有项目配置，不会修改"
    → 询问是否需要补充件（STACK.md / workflow/ 目录）
    → 只生成用户同意的补充件，绝不动 CLAUDE.md

(C) 有 CC-GEB 生成的 CLAUDE.md
    → 告知当前级别，建议 upgrade 或 check

(D) 有 CLAUDE.md + 有 .claude/ 项目配置
    → 同 (B)，尊重已有配置
```

**判断 CC-GEB 生成：** CLAUDE.md 含 `<!-- Generated by CC-GEB -->` 或 `GEB Loop` 字样。

### Step 3: 推荐档位

根据项目规模推荐：

```
空目录 或 <5 个源文件     → 推荐轻量
5-30 个源文件             → 推荐标准
30+ 个源文件 或 monorepo  → 推荐完整
```

向用户展示：

```
? 选择配置级别:

  轻量 — 生成项目描述，不加文档约束
         适合：开源项目、小脚本、快速修 bug

  标准 — + 技术栈 + GEB 文档同步（只约束 Claude 改动的文件）
         适合：有持续迭代的中型项目

  完整 — + workflow 六层 + 回写闭环 + 设计哲学
         适合：大型产品、长期维护的核心项目
```

### Step 4: 智能生成

读取对应模板（`~/.claude/templates/light.md` / `standard.md` / `full.md`），**AI 智能填充所有字段**：

- 从 package.json/go.mod 等读取项目名和依赖
- 扫描目录结构填充 `<directory>`
- 从 README.md 或代码结构生成项目概述
- **不留任何 {placeholder}，全部由 AI 填好**

生成文件：

| 级别 | 生成内容 |
|------|---------|
| 轻量 | `CLAUDE.md` |
| 标准 | `CLAUDE.md` + `STACK.md` |
| 完整 | `CLAUDE.md` + `STACK.md` + `workflow/` 六层目录 |

CLAUDE.md 顶部加标记：`<!-- Generated by CC-GEB /cc-geb init -->`

### Step 5: workflow/ 目录（仅完整模式）

```
workflow/
├── 00-overview/README.md     项目索引
├── 01-rule/README.md         元规则
├── 02-session/README.md      初始化记录（写入本次事实）
├── 03-design/README.md       设计层入口
├── 04-testing/README.md      验证层入口
├── 05-execution/README.md    执行层入口
└── 06-knowledge/README.md    知识层入口
```

### Step 6: 确认

- 列出所有新生成的文件
- 提示用户审阅项目描述是否准确
- 建议 git add

### 补充件模式（场景 B/D）

已有非 CC-GEB 的 CLAUDE.md 时，逐个询问：
1. "要不要生成 STACK.md？"
2. "要不要生成 workflow/ 目录？"

每个独立生成，用户可以只选其一或都不要。**绝不动已有 CLAUDE.md。**

## Check：文档与代码一致性巡检

### 执行流程

1. **定位项目根目录**（git root 或 pwd）

2. **扫描文档清单**
   - CLAUDE.md（任意格式）
   - README.md
   - STACK.md（如有）
   - docs/ 目录

3. **代码结构 vs 文档对比**
   - 扫描顶层目录和 src/ 下的模块
   - 对比 CLAUDE.md 中记录的目录列表
   - 找出：幽灵文档、隐形模块、陈旧文档

4. **依赖一致性**
   - 读取包管理文件的实际依赖
   - 对比 STACK.md 声明（如有）
   - 找出未记录的新依赖 / 已移除的旧依赖

5. **生成报告**

```
CC-GEB Check 报告 — {project_name}
──────────────────────────────────────

健康项：
  - CLAUDE.md 存在且最近更新
  - 目录结构与文档一致

需要关注：
  - [幽灵] CLAUDE.md 提到 utils/ 但目录不存在
  - [隐形] src/middleware/ 存在但文档未记录
  - [陈旧] docs/api.md 超过 60 天未更新
  - [漂移] package.json 有 zod 但 STACK.md 未声明

建议操作：
  1. 从 CLAUDE.md 移除 utils/ 引用
  2. 在 CLAUDE.md 中添加 src/middleware/ 描述
```

6. **可选自动修复**：用户确认后执行建议操作

### 关键约束
- 只读为主，修改需用户确认
- 不扫描 node_modules、dist、build、.git
- 不要求 CC-GEB 格式，任何项目都可用

---

## Upgrade：升级配置级别

### 执行流程

1. 读取当前 CLAUDE.md，判断当前级别
2. 展示可升级选项：
   - 轻量 → 标准：追加 GEB Loop 规则 + 生成 STACK.md
   - 标准 → 完整：追加设计哲学 + 生成 workflow/ 目录
   - 轻量 → 完整：一步到位
3. 用户选择后，**在已有 CLAUDE.md 尾部追加新内容**，不改动已有部分
4. 生成新增的配套文件

### 关键约束
- 只追加，不修改已有内容
- 只对 CC-GEB 生成的 CLAUDE.md 执行升级
- 非 CC-GEB 的 CLAUDE.md → 提示"请先用 /cc-geb init 初始化"

---

## 全局约束

- **绝不覆盖已有的 CLAUDE.md**（非 CC-GEB 生成的）
- **绝不覆盖已有的 STACK.md**
- 模板中的 placeholder 必须被 AI 智能填充，不留给用户
- Check 功能不要求 CC-GEB 格式，任何项目都可用
